## Context

Парсер уже сохраняет `^TYPE` в `meta.type_annotation` для атомов и списков. `fn_macro` понижает `fn` в `fn*`, Java-компилятор уже понимает core-форму `(cast TYPE value)`, но JavaScript и интерпретатор её не обрабатывают. Простое добавление `(let* [x (cast TYPE x)] ...)` недопустимо для Java: оно повторно объявит параметр `x` в области метода или лямбды.

## Goals / Non-Goals

**Goals:**

- Позволить писать `^TYPE` на символьных параметрах `fn` и `defn`.
- Предоставить телу функции исходное имя параметра, уже приведённое к объявленному типу.
- Сохранить выполнение одного исходника в `eval`, JavaScript и Java.
- Использовать существующие core-формы и `Gensym` без новых AST-типов или runtime-зависимостей.

**Non-Goals:**

- Проверять типы во время выполнения для `eval` или JavaScript.
- Поддерживать аннотации на destructuring-параметрах.
- Добавлять статическую проверку корректности Java-типов до `javac`.

## Decisions

### Переименять аннотированный формальный параметр

Для каждого символьного параметра `^T x` макрос создаёт свежий фактический параметр через `Gensym.gensym` и initial `let*` binding с исходным именем:

```clojure
(fn [^java.util.List xs] body)
;; =>
(fn* [G__1] (let* [xs (cast java.util.List G__1)] body))
```

Это сохраняет лексическую область имени `xs` для тела и создаёт валидный Java-код. Альтернатива с binding `xs` из `xs` отклонена: Java запрещает повторно объявлять параметр метода или лямбды. Переписывание всех ссылок тела на fresh name отклонено, поскольку потребовало бы отдельного анализа областей видимости.

### Использовать один initial `let*` для всех аннотированных параметров

Макрос сохраняет порядок и арность исходных параметров; неаннотированные параметры остаются формальными. Для всех аннотированных параметров создаётся один `let*` перед исходным телом. Это использует уже поддерживаемую форму последовательных bindings и не требует изменений в lowering destructuring.

### Сделать `cast` прозрачной core-формой вне Java

Интерпретатор возвращает значение `(cast TYPE value)`, вычисляя только `value`. JavaScript генерирует выражение для `value` и не включает `TYPE` в output. Java сохраняет существующую генерацию `(TYPE) value`. Альтернатива с Java-only параметрическими аннотациями отклонена: один и тот же исходник перестал бы работать для `eval` и JS после дешугорирования.

### Ограничить первую версию символьными параметрами

`^T [a b]` и другие аннотации на destructuring-паттернах не получают семантики в этом изменении. Для них не определено, требуется ли приводить весь аргумент или отдельные leaf bindings; это можно добавить отдельным изменением.

## Risks / Trade-offs

- [No-op cast скрывает неверный тип на `eval` и JavaScript] → Это намеренная переносимая семантика; фактическую проверку выполняет Java при компиляции или запуске.
- [Fresh имя может быть видно в сгенерированном коде] → Используется существующий `Gensym.gensym`, уже применяемый lowering и макросами для предотвращения коллизий.
- [Новый core-вызов может быть ошибочно скомпилирован как обычная функция] → Добавить специальные ветви `cast` до общего вызова в `eval` и JS-компиляторе.
