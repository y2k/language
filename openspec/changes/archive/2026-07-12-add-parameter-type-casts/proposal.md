## Why

Java-интероп требует явных приведений аргументов, но повторять `(cast T x)` в теле каждой функции шумно и не гарантирует, что всё тело использует приведённое локальное имя. Аннотация типа параметра должна объявлять это приведение один раз, не делая исходный код непереносимым на `eval` и JavaScript.

## What Changes

- Аннотация `^TYPE` на символьном параметре `fn` или `defn` будет дешугорироваться в локальный `(cast TYPE ...)` до выполнения тела функции.
- Дешугорирование будет использовать свежий входной параметр, чтобы Java-код не переобъявлял имя параметра в той же области видимости.
- `cast` станет no-op формой для целей `eval` и JavaScript, сохраняя значение без проверки типа.
- Java сохранит генерацию настоящего Java cast для формы `cast`.
- Аннотации на destructuring-параметрах не входят в объём изменения.

## Capabilities

### New Capabilities

Нет.

### Modified Capabilities

- `frontend-syntax`: аннотации типов символьных параметров функций дешугорируются в локальные cast bindings.
- `compiler-targets`: JavaScript поддерживает `cast` как no-op, а Java-компиляция параметрических аннотаций использует локальные casts.
- `eval-runtime`: интерпретатор поддерживает `cast` как no-op форму.

## Impact

Затрагиваются `frontend/builtin_macros.ml`, `backend_compiler/java.ml`, `backend_compiler/js.ml`, `backend_eval/eval.ml` и соответствующие frontend, Java, JavaScript и eval тесты. Публичный синтаксис расширяется `^TYPE` на простых параметрах; новые зависимости не требуются.
